## 三、核心工作流

### 首次分析用户

```
① 并行拉取 6 组数据
   ├── fitness（VO2max / 预估成绩 / 阈值配速/心率）
   ├── health（每日健康汇总）
   ├── records（运动记录列表）
   ├── recovery（恢复状态）
   ├── load（训练负荷）
   └── hrv（HRV 评估）
② 提取关键字段 → 确定用户基础体能
③ 拉取近 K 天天气概览（分月/季）
④ 输出分析结论
```

### 单次训练分析

```
① 调用 getActivityDetail 获取 MCP 汇总
② 判断是否需要网页端细化？
   ├── 需要（间歇/晨跑夜跑/时段天气/多次训练）
   │   ├── 有授权 → fetchActivityDetail() 获取精确数据
   │   │              ├── startTime.period → 判定晨跑/夜跑
   │   │              ├── weather → 精确温湿度+风
   │   │              └── tables[0] → 计圈表
   │   └── 无授权 → 询问用户 → 同意则引导授权，拒绝则降级
   └── 不需要（普通慢跑/长距离汇总）→ 跳过
③ 天气数据（优先级：网页端精确实测 > Open-Meteo 全天均值）
④ 查前3天跑量和睡眠（疲劳因素，优先级最高）
⑤ 综合解读：疲劳 > 天气 > 训练意图
```

## 四、关键规则（必须记住）

### ⚠️ 4.1 睡眠日期错位（高驰 API 特性）

`querySleepData` 返回的日期 = **醒来当天**，不是入睡当天。

| API 返回 | 实际入睡 |
|---------|---------|
| M/D | (M/(D-1)) 晚 |
| N 日 | (N-1) 日 |

📌 查"N晚"睡眠 → endDate 设为 N+1。

### ⚠️ 4.2 运动记录无时间戳

`querySportRecords` 的日期字段不含时分秒，记录顺序不保证时间顺序。
- ❌ 不要推断上午/下午
- ✅ 同一天多次运动 → 询问用户确认
- ✅ 需要精确时间 → 走网页端细化（`time-weather-refinement.md`）

### ⚠️ 4.3 分析优先级

**疲劳 > 天气 > 训练意图**

查天气前，**先查前3天跑量和睡眠**——这是最常见的分析错误。

### 📁 4.4 文件输出规则

| 规则 | 说明 |
|------|------|
| 默认目录 | `{workspace}/高驰数据分析/` |
| 禁止 | 往 C 盘写文件、重新拉数据 |
| 格式 | HTML，CSS 内联 |
| 文件名 | `运动数据分析报告_YYYYMMDD.html` |
| 流程 | 对话框先出结论 → 末尾询问是否保存 |

## 六、调试要点

| 现象 | 怎么做 |
|------|--------|
| 数据疑似不对 | 先检查工具 inputSchema（参数名/结构） |
| 需要交叉验证 | 用 `queryDailyHealthData`（最稳定） |
| 多次请求结果完全相同 | 参数名可能写错了（MCP 缓存了旧结果） |
| 登录态过期 | 运行 `coros_web_login.js` 重新授权 |
