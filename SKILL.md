## 三、核心工作流

### 首次分析用户

```
① 并行拉取 6 组数据
   ├── fitness（VO2max / 预估成绩 / 阈值配速/心率）
   ├── health（每日健康汇总）
   ├── records（运动记录列表）
   ├── recovery（恢复状态）
   ├── load（训练负荷）
   └── hrv（HRV 评估）
② 提取关键字段 → 确定用户基础体能
③ 拉取近 K 天天气概览（分月/季）
④ 输出分析结论
```

### 单次训练分析

```
① 调用 getActivityDetail 获取 MCP 汇总
② 判断是否需要网页端细化？
   ├── 需要（间歇/晨跑夜跑/时段天气/多次训练）
   │   ├── 有授权 → fetchActivityDetail() 获取精确数据
   │   │              ├── startTime.period → 判定晨跑/夜跑
   │   │              ├── weather → 精确温湿度+风
   │   │              └── tables[0] → 计圈表
   │   └── 无授权 → 询问用户 → 同意则引导授权，拒绝则降级
   └── 不需要（普通慢跑/长距离汇总）→ 跳过
③ 天气数据（优先级：网页端精确实测 > Open-Meteo 全天均值）
④ 查前3天跑量和睡眠（疲劳因素，优先级最高）
⑤ 综合解读：疲劳 > 天气 > 训练意图
```

## 四、关键规则（必须记住）

### ⚠️ 4.1 睡眠日期错位（高驰 API 特性）

`querySleepData` 返回的日期 = **醒来当天**，不是入睡当天。

| API 返回 | 实际入睡 |
|---------|---------|
| M/D | (M/(D-1)) 晚 |
| N 日 | (N-1) 日 |

📌 查"N晚"睡眠 → endDate 设为 N+1。

### ⚠️ 4.2 运动记录无时间戳

`querySportRecords` 的日期字段不含时分秒，记录顺序不保证时间顺序。
- ❌ 不要推断上午/下午
- ✅ 同一天多次运动 → 询问用户确认
- ✅ 需要精确时间 → 走网页端细化（`time-weather-refinement.md`）

### ⚠️ 4.3 分析优先级

**疲劳 > 天气 > 训练意图**

查天气前，**先查前3天跑量和睡眠**——这是最常见的分析错误。

### 📁 4.4 文件输出规则

| 规则 | 说明 |

| 默认目录 | `{workspace}/高驰数据分析/` |

## 七、历史教训（血泪总结）

> 🚨 **以下所有教训来自用户实际纠正，必须严格遵守，再犯用户会发火。**

### 7.1 配速表达规则

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 用800m完成时间表达配速（"3分跑800m"） | **统一用 min/km 配速**（4:35/km），不要让用户换算 |
| 说"最快单圈3:45"没注明口径 | 标注来源：APP口径 vs MCP口径 vs 网页端口径 |

### 7.2 心率判断规则

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 用"均心率146"判断间歇训练质量好坏 | **间歇训练质量用"最快单圈配速"作为代理指标**，不能单凭均心率 |
| 认为 MAF 144bpm = 马配心率 | **MAF 是有氧上限（130-144bpm），马配心率由 LT 决定**，用户全马心率160-164bpm |
| 瞎编"马配心率148-155" | **马配心率 = 用户个人档案中登记的值**，不确认就老实说"数据不足" |

### 7.3 睡眠归因规则（重要！）

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 把6/10的深睡差归因到6/9的间歇（20:09跑） | **必须核对运动开始时间**，20:00左右跑完不影响睡眠，22:00后跑才会 |
| 不区分晚跑时间就下睡眠结论 | 用网页端 `fetchActivityDetail` 获取精确 `startTime.period` 和 `startTime.time`，**20:00前 vs 22:00后 对睡眠影响完全不同** |

**睡眠归因核对表：**
```
找"训练日当晚睡眠差"的原因时：
  ① 查该次训练的精确开始时间（网页端 startTime）
  ② 如果 ≤21:00 → 睡眠差不是该训练直接造成的（查其他原因）
  ③ 如果 ≥21:30 → 可能是训练/兴奋影响睡眠
```

### 7.4 数据引用规则

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 说"6/11可能是力量训练"（猜测） | **有数据就引用数据，没有就说"无记录"**。6/11有健走记录(1.45km)就直接说健走 |
| "您这状态打全马3:20以内完全有戏"（过度承诺） | 训练建议要基于数据说话，少用主观判断 |

### 7.5 训练负荷分析规则

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 建议跑休后用户实际跑了10km（4:35/km） | **间歇次日建议跑休时，如果用户有其他安排，尊重用户体感**，不做事后批评 |
| 说"恢复跑5-6km"后用户跑了10km | 先确认用户是否有其他计划，再给建议 |

### 7.6 乳酸相关表述（⚠️ 犯过两次，别再犯了）

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 说"清乳酸" | ❌ **不准出现"排酸""清乳酸""代谢废物"等任何字眼**。乳酸在运动后1-2小时自然清除，不需要任何操作 |
| 说"半马后肌肉里堆积的代谢废物，躺着清不如慢跑清" | ✅ 正确说法：**"主动恢复（Active Recovery）——轻度活动促进血液循环，加速氧气和营养输送到疲劳肌肉，帮助组织修复"** |
| 正确表述 | "促进腿部血流，加速恢复" / "主动恢复促进循环和组织修复" |

### 7.7 个人记录恢复

高驰 App 个人记录误操作恢复流程（非"上古时期"方法）：
1. 打开 App → **「我」→「个人记录」**
2. **左滑**错误的记录行 → 点击 **「忽略」**
3. 系统自动从剩余记录中重算 PB
4. 如找不到目标记录 → 去 **「所有运动记录」** 列表查找对应日期的记录
5. 进入记录详情页，查找「忽略」开关并关闭
6. 如都不行 → 联系 COROS 客服（**「我」→「设置」→「帮助与反馈」**）

### 7.8 周跑量 & 日期计算规则

| ❌ 之前犯的错 | ✅ 正确做法 |
|:-------------|:-----------:|
| 建议周跑量47km，用户说至少60k | **用户周跑量基线60km+**，课表设计以此为前提 |
| 只排了周一~周六，漏了周日 | **一周是周一~周日，7天**，周日必须出现 |
| 赛后第二天建议"排酸跑" | 赛后放松跑 = 主动恢复（Active Recovery），理由见7.6 |
